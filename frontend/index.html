01

Storyline

The motivation and vision behind this DBMS project

The Problem

Campus placement processes at universities are typically handled through spreadsheets, emails, and manual coordination. This leads to data inconsistency, missed deadlines, and difficulty matching eligible students to the right companies.

Our Solution

We designed a relational database system — CampusPlacementPortal — that digitises the entire recruitment lifecycle. Student profiles, job postings, applications, and interview outcomes are tracked in one interconnected MySQL database.

Real-World Impact

Placement coordinators can see which student applied where, filter candidates by CGPA, track interview schedules, and view statistics in real time through a live web interface backed by a REST API.

Components of Database Design

Building blocks used to design the CampusPlacementPortal database

Entities (Tables)

  • Students — Academic profiles of candidates
  • Companies — Recruiting organisations
  • Job_Postings — Open roles with eligibility criteria
  • Applications — Student–job linkage records
  • Interviews — Scheduled interview outcomes

Attributes

  • Simple — name, email, cgpa
  • Multi-valued — skills (comma-separated)
  • Optional (NULL) — resume_link, skills
  • Derived — placement year from graduation_year
  • Composite — location (city + state)

Keys

  • PK AUTO_INCREMENT INT — student_id, company_id…
  • FK company_id in Job_Postings → Companies
  • FK student_id, job_id in Applications
  • UQ email, hr_email, company_name
  • CHK cgpa BETWEEN 0 AND 10

Relationships

  • 1 : N — One company posts many jobs
  • 1 : N — One student submits many applications
  • 1 : N — One job receives many applications
  • 1 : 1 — One application has one interview record

Integrity Constraints

  • Entity Integrity — PKs are always NOT NULL
  • Referential Integrity — FK must reference valid PK
  • Domain Integrity — CHECK on cgpa, min_cgpa
  • Default — status DEFAULT 'Applied'

Design Principles

  • Single Responsibility — Each table = one entity
  • Normalisation — Up to 3NF to remove redundancy
  • Scalability — AUTO_INCREMENT PKs for growth
  • JOINs over duplication — FK references instead of copying data

Normalization

Analysis of normal forms to eliminate redundancy and ensure data integrity

1NF

First Normal Form

All attributes must contain atomic values; no repeating groups.

✅ Each column holds a single atomic value
✅ Every table has a defined primary key
✅ No repeating groups or arrays exist in any column
📝 skills uses comma-separated values — a design trade-off
✅ All tables satisfy 1NF
2NF

Second Normal Form

Must be in 1NF. All non-key attributes must be fully dependent on the entire primary key.

✅ All primary keys are single-column (no composite PKs)
✅ No partial dependencies can exist with single-column PKs
✅ role, package_lpa fully depend only on job_id
✅ status, applied_date fully depend only on application_id
✅ All tables satisfy 2NF
3NF

Third Normal Form

Must be in 2NF. No non-key attribute may transitively depend on the primary key.

✅ company_name is NOT stored in Job_Postings — only company_id (FK)
✅ Student details are NOT stored in Applications — only student_id (FK)
✅ Denormalized data is retrieved at query time via JOINs
✅ No A → B → C transitive dependency chain exists
✅ All tables satisfy 3NF

SQL Queries

Key SQL operations demonstrated in this project

DDL — CREATE TABLE
CREATE TABLE Students(
  student_id INT AUTO_INCREMENT PRIMARY KEY,
  name       VARCHAR(50) NOT NULL,
  email      VARCHAR(100) UNIQUE NOT NULL,
  branch     VARCHAR(50) NOT NULL,
  cgpa       DECIMAL(3,2) CHECK (cgpa BETWEEN 0 AND 10),
  graduation_year INT NOT NULL,
  skills     VARCHAR(200),
  resume_link VARCHAR(200)
);
DML — SELECT with JOIN
SELECT s.name, c.company_name, j.role
FROM Students s
JOIN Applications a ON s.student_id = a.student_id
JOIN Job_Postings j ON a.job_id = j.job_id
JOIN Companies c   ON j.company_id = c.company_id;
DML — INSERT
INSERT INTO Students(name,email,branch,cgpa,graduation_year)
VALUES ('Vikas Kumar','vikas@gmail.com','CSE',8.1,2025);
DML — UPDATE
UPDATE Students
SET cgpa = 8.8
WHERE student_id = 1;
DML — DELETE
DELETE FROM Students
WHERE student_id = 7;
Aggregate Functions
SELECT AVG(cgpa)       AS Average_CGPA   FROM Students;
SELECT COUNT(*)        AS Total_Students FROM Students;
SELECT MAX(package_lpa) AS Highest_Package
FROM Job_Postings;
GROUP BY & HAVING
SELECT branch, COUNT(student_id) AS Total_Students
FROM Students
GROUP BY branch
HAVING COUNT(student_id) > 2;
Subquery
-- Students with CGPA above average
SELECT name FROM Students
WHERE cgpa > (
    SELECT AVG(cgpa) FROM Students
);
Nested Query
-- Students eligible for Google
SELECT name FROM Students WHERE cgpa >= (
  SELECT min_cgpa FROM Job_Postings
  WHERE company_id = (
    SELECT company_id FROM Companies
    WHERE company_name = 'Google'
  )
);
Dashboard Stats
SELECT
  (SELECT COUNT(*) FROM Students)    AS total_students,
  (SELECT COUNT(*) FROM Companies)   AS total_companies,
  (SELECT COUNT(*) FROM Job_Postings) AS total_jobs,
  (SELECT AVG(cgpa) FROM Students)   AS avg_cgpa,
  (SELECT MAX(package_lpa) FROM Job_Postings) AS max_package;

Learning from the Project

Key insights gained by building and deploying this system end-to-end

01

Referential Integrity is Non-Negotiable

We learned first-hand why foreign key constraints matter — attempting to delete a company that has jobs linked to it throws a referential integrity error. This enforces data safety automatically.

02

Normalisation Prevents Real Problems

By separating company info from job postings, we ensured a company's name update propagates everywhere automatically — there's no stale data to fix manually.

03

ER Diagrams Are Planning Tools

Designing the ER diagram before writing any code helped us identify the Applications table as a junction entity and avoid incorrect table structures.

04

JOINs Bring Data to Life

A single well-written JOIN query across Students → Applications → Job_Postings → Companies can replace four separate queries and four client-side merge operations.

05

API Design Affects Frontend Complexity

A poorly designed API response forces the frontend to do expensive look-up work. Good backend JOINs return complete data and make the frontend simple and fast.

06

CHECK Constraints Are Underused

Using CHECK (cgpa BETWEEN 0 AND 10) at the database level eliminates the need for this validation in both backend and frontend — one point of truth.

10

Challenges Faced

Obstacles encountered during development and how we overcame them

01

Foreign Key Delete Errors

Problem: Deleting a student who has active applications caused a referential integrity violation.

Solution: We handle these errors gracefully in the API and inform the user to delete linked applications first.

02

Syncing Frontend Dropdowns

Problem: After adding a new company, the "Select Company" dropdown in the job form did not update automatically.

Solution: We call loadAllData() after every POST operation, which re-fetches and re-renders all dropdowns.

03

CORS Error on First Run

Problem: Browser blocked fetch requests from frontend (port 5500) to backend (port 3000) due to CORS policy.

Solution: Added the cors middleware to Express and served the frontend as a static directory from the same port.

04

Designing the ER Diagram

Problem: Determining the correct cardinality between Applications and Interviews (1:1 vs 1:N) was initially unclear.

Solution: Through analysis of real placement scenarios we confirmed one application should yield at most one interview record in our scope.

05

MySQL Connection from Node.js

Problem: Initial connection attempts to MySQL failed due to authentication plugin mismatch (caching_sha2_password vs mysql_native_password).

Solution: Updated MySQL user authentication method using ALTER USER command in MySQL Workbench.

06

Keeping Data Consistent Across Tabs

Problem: Adding a student in the Students tab did not update the Applications tab dropdown until page refresh.

Solution: Centralised all state into module-level arrays and re-render all panels after every data mutation.

Conclusion

What we built, what we achieved, and the way forward

Project Summary

The University Campus Placement & Internship Portal is a fully functional, end-to-end web application backed by a normalised MySQL relational database. It demonstrates the real-world application of core DBMS concepts including entity-relationship modelling, schema design, normalisation up to 3NF, SQL DDL/DML operations, JOINs, subqueries, and aggregate functions.

Database

5 normalised tables, 3NF compliant, with proper PK/FK constraints, CHECK constraints, and default values.

Backend API

10+ RESTful endpoints built with Node.js and Express.js serving live CRUD operations over MySQL.

Frontend

Single-page app with 11 documented sections and a live demo panel — built with pure HTML, CSS, and JavaScript.
